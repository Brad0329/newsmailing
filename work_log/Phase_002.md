# Phase 002 — Naver 검색 API 연동

**완료일**: 2026-04-17
**상태**: 테스트 통과 (18/18 단위 테스트 + 실제 API 스모크 PASS)

## 결정의 이유

- **`sort=sim` 고정 (정확도순)**: 사용자와 사전 합의한 A안. 단순/정확/무비용. 관련성이 아쉬울 경우 Phase 006 선택적 확장에서 LLM 랭킹으로 보강.
- **`display=30` 기본값**: Naver 최대치(100)는 호출 한도 낭비. 어제/오늘 필터 후 키워드당 3~5개 선별에 30개면 충분히 후보 확보.
- **"어제/오늘" 정의**: `(어제 00:00 KST) ~ (현재 KST)`. 자정 직후 실행해도 어제 전체가 포함되며, 미래 타임스탬프(시차/오타)는 제외.
- **KST 고정**: 서버가 어느 시간대에 있든 한국 독자 대상이므로 한국 기준 어제/오늘로 계산해야 함. 서버 OS 타임존 설정에 의존하지 않도록 `timezone(timedelta(hours=9))`를 상수로 내장.
- **`Article` dataclass**: dict보다 타입 힌트/IDE 지원이 낫고, 필요 시 `to_dict()`로 JSON 직렬화.
- **출처는 도메인 매핑으로 추출**: Naver API가 언론사명을 안 줌. 전체 언론사 매핑은 불가능하므로 주요 30여 개만 초기 등록, 매핑 없으면 도메인 그대로 노출 (UX상 도메인도 출처 식별에는 충분).
- **전역 중복 제거 (URL 기준)**: 키워드 여러 개가 같은 기사를 히트하는 경우 흔함. "같은 기사를 여러 번 보낸다"는 UX 사고를 방지.

## 외부 제약 조건 (운영 메모)

- **Naver 검색 API 헤더**: `X-Naver-Client-Id`, `X-Naver-Client-Secret` (Bearer 아님. 오타 주의).
- **호출 한도**: 25,000/일 (무료 플랜). 20 키워드 × 일 1회 = 20 — 여유 충분.
- **응답 내 `<b>` 태그와 HTML 엔티티**: Naver가 검색어 하이라이트 목적으로 삽입. `strip_html()`에서 반드시 정리 (그대로 메일에 넣으면 태그가 보임).
- **`originallink` vs `link`**: `link`는 naver.com 리다이렉트인 경우가 많음. 독자가 원 기사로 바로 가도록 `originallink` 우선 사용. 단, 이게 없으면 `link`로 폴백.
- **일자 필터는 클라이언트에서**: Naver API에 `pubDate` 범위 파라미터가 없다. `sort=date`로 받고 직접 잘라내거나, `sort=sim`으로 받고 어제/오늘만 남기거나 선택 — 후자 채택 (정확도 우선이라).

## 향후 운영상 주의점

- **도메인 매핑 누락**: 실제 운영 시작 후 "출처가 도메인으로 표시되는 기사"가 자주 보이면 `DOMAIN_TO_SOURCE`에 추가. Phase 005 마감 시 상위 30~50개로 확장 예정.
- **Windows 콘솔 한글 표시**: 테스트 agent 리포트에서 언급 — cp949 기본 콘솔에서 직접 print 시 깨질 수 있음. 애플리케이션 자체는 UTF-8로 처리하므로 HTTP/HTML 경로에 영향 없음. 스크립트로 디버그할 때만 `PYTHONIOENCODING=utf-8` 설정 권장.
- **서브도메인 제거 정규식**: `www|news|biz|sports|m|imnews|v` — 실제 운영에서 누락 서브도메인이 발견되면 추가 필요 (예: `today.`, `go.`). 단, 너무 공격적으로 제거하면 별개 언론사를 같은 도메인으로 묶을 위험이 있으므로 신중히.
- **API 429 (rate limit) 미처리**: 현재 `raise_for_status()`만 호출. 20 키워드 정도로는 실측 이슈 없으나, 향후 키워드가 100+ 수준으로 늘면 재시도/지연 로직 필요.
- **네트워크 타임아웃 10초 고정**: 사내 프록시 환경에서 더 길게 조정 필요할 수 있음.
