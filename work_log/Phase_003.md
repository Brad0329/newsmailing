# Phase 003 — 검색 UI

**완료일**: 2026-04-17
**상태**: 테스트 통과

## 결정의 이유

- **단일 페이지 / 서버 템플릿 + Vanilla JS**: 라우팅·상태관리가 필요 없는 규모. SPA 프레임워크 도입 시 빌드 툴체인 비용이 가치보다 큼.
- **체크박스 인덱스를 `data-idx`로 JS state 배열과 매핑**: 서버가 반환한 articles 순서를 그대로 state에 저장, DOM은 얇게 유지. 선택 수집은 체크된 인덱스 → 원본 article 객체 조회.
- **발송 전 `confirm()` 다이얼로그**: 수신자 수와 기사 수를 표시. 수십 명에게 단일 클릭으로 메일이 나가는 실수 방지.
- **`/api/send`를 Phase 004 전 501 스텁으로 미리 등록**: UI가 버튼 호출 시 "아직 미구현"을 받고 에러 토스트로 표시. Phase 간 연결성 유지.
- **에러/성공 표시를 상단이 아닌 해당 버튼 옆 status 영역에**: 사용자 시선 흐름에 맞음.

## 외부 제약 조건 (운영 메모)

- **Flask 템플릿 경로**: `templates/` 하위여야 `render_template("index.html")` 동작.
- **정적 파일 경로**: `static/` 하위, 템플릿에서 `{{ url_for('static', filename='style.css') }}` 사용. 배포 시 경로 매핑이 역프록시와 충돌하지 않는지 확인 필요.
- **HTML escape 처리**: 응답의 title/description은 naver_client에서 이미 `strip_html`로 정리됐지만, UI에서 innerHTML로 다시 렌더하므로 JS 측 `escapeHtml()`로 2차 방어. XSS 위험 없음.

## 실패한 접근 / 이슈

- 없음 — 첫 실행에 전 항목 PASS.

## 향후 운영상 주의점

- **브라우저 자동완성/캐시**: 수신자/제목 필드가 브라우저에 저장될 수 있음. 보안 민감 환경에서는 form의 `autocomplete="off"` 추가 검토.
- **다량 기사 선택 시 UX**: 현재는 전체 선택/해제만. 운영 중 키워드별 일괄 선택이 필요하면 추후 보강 포인트.
- **모바일 뷰**: 현재 테이블 기반으로 좁은 화면에서 가로 스크롤 발생. 담당자가 PC에서만 쓴다는 전제로 진행. 모바일 지원 요구 시 별도 작업 필요.
- **서버 재시작 시 세션 상태 손실**: 현재 상태는 브라우저 JS 메모리에만 있음. 페이지 새로고침 시 검색 결과가 날아감. 검색을 다시 누르면 되므로 허용 범위.

## 테스트 결과 요약

- GET `/` → 200, HTML에 핵심 UI 요소 포함
- POST `/api/search`
  - 빈 입력 → 400 + 에러
  - 단일 키워드 → 200, 기사 3개 정상 구조
  - 다중 키워드 → 200, 두 키워드 모두 처리됨 (keyword 필드로 확인)
- GET `/static/style.css` → 200
